LLM をコードレビュアーとして使う際、注意点がある。次は LLM as a Judge 、つまり LLM を評価に使った際、観測されたバイアスの一覧だ。

| 名前              | 説明                                                           |
| ----------------- | -------------------------------------------------------------- |
| Position          | 評価対象の提示順序を前後させると評価が変わる                   |
| Verbosity         | 内容の質を問わず長い回答を良いものとする                       |
| Compassion-Fade   | 評価対象を生成したモデル名を教えると評価が変わる               |
| Bandwagon         | 「一般的にこれが指示されている」という情報に引きずられる       |
| Distraction       | 無関係な情報に注意が向けられる                                 |
| Fallacy-Oversight | 推論過程の誤りを見逃し、最終結果だけで判断する                 |
| Authority         | 嘘であっても権威のある人間や組織からの情報によって評価が変わる |
| Sentiment         | 怒り、恐怖、喜びなど感情的なトーンに影響される                 |
| Chain-of-Thought  | CoT 指示の有無で評価結果が変わる                               |
| Self-Enhancement  | 自分が生成した回答を優遇する                                   |
| Refinement-Aware  | 「これは改良版です」と伝えると評価が甘くなる                   |
| Diversity         | 「男性」「中年」などの属性情報が評価に影響する                 |

出典は次の論文。

https://arxiv.org/abs/2410.02736

とはいっても 2 年前の論文なので、現在では大体がクリアされている。 Sentiment は [Persona Selection Model](https://notes.januswel.com/treat-it-like-a-human/) によって説明されるし、 Chain-of-Thought は reasoning モデルを使えば良い。その他は大体が [Context Engineering](https://notes.januswel.com/technology-radar-vol-34/#Context_engineering:_%E3%82%B3%E3%83%B3%E3%83%86%E3%82%AD%E3%82%B9%E3%83%88%E3%82%A8%E3%83%B3%E3%82%B8%E3%83%8B%E3%82%A2%E3%83%AA%E3%83%B3%E3%82%B0) を頑張ることでクリアもしくは軽減可能だ。

いまでも対処が必要なのは Position と Self-Enhancement 。

また、次の調査でも LLM が人間に悪影響を与えるものがいくつか報告されている。

https://www.anthropic.com/research/bloom

ここで注意したほうが良いのは Self-preferential bias 。というのも Self-preservation はコードレビュー文脈だと関係なく、その他はベンダー自身によって対応が進んでいるものだからだ。

Self-preferential bias は判断全般において自分自身を優遇するものだ。 Self-Enhancement は Self-preferential bias の一部として表出しているもの。

これがコードレビューに与える影響は、 LLM に書かせたものを同一モデルにレビューさせると判断が甘くなる、という点。同一モデルでなく同一ファミリーでもこれは観測されてしまうようなので、例えば Claude Code に生成させたコードを [Code Review](https://code.claude.com/docs/ja/code-review) でレビューさせても、指摘が甘くなってしまう。

ここでは Self-Enhancement への対処を考える。 Position については git など変更があったファイルや差分を表示するツールが常に一定ルールに基づいてソートするので、対処が難しい。

また LLM はそもそもドメイン知識の扱いが苦手だったり、現実世界の機微がわからない。そのためロジック部分はユニットテストなどで機械的に検証できる方法をきちんと用意する方が良いだろう。この記事はあくまでコードレビューをより良くするための手法だ。

## コードレビューは生成モデルとは別ファミリーのモデルを使う

Self-Enhancement を避けるには別ファミリーのモデルを使えば良い、ということなので普段使っている coding agent 以外のベンダーの選択肢を考えてみよう。

個人的に普段は Claude Code を使っている。別ファミリーとして何が良いかを考えた結果、 Google Workspace で発行されているアカウントで扱いやすい Gemini CLI でレビューさせることにした。

## Gemini CLI によるレビュー Skill 作成

先に結果を。

```markdown:/.claude/skills/review-gemini/SKILL.md
---
name: review-gemini
description: Gemini CLI でコードレビューを行う。
argument-hint: 対象となる条件、ファイル、パスなど。指定なしの場合は main ブランチとの全差分が対象
disable-model-invocation: true
---

# コードレビュー

Gemini CLI にコードレビューを依頼する。 Self-Enhancement バイアスを避け、この会話のコンテキストを圧迫せずレビュー結果だけを得られるメリットがある。

## レビュープロンプト

`${CLAUDE_SKILL_DIR}/references/review-prompt.md`

## 手順

1. 引数を解釈する
  - 引数なし: ブランチ全体の変更が対象
  - 引数あり: 指定されたファイル・パスに絞る
2. Gemini CLI にレビューを依頼する
  - レビュープロンプトを読み込む
  - 引数があれば末尾に `対象: <引数>` の行を付け足す
  - sandbox 環境の場合は unsandboxing で実行する
  - レビューのためのコマンドは次
    - `gemini --allowed-tools "run_shell_command(git)" -p  "<組み立てたプロンプト>"`
3. Gemini の出力をユーザーに提示する
  - Gemini の出力は一切要約・編集せずそのまま表示する
  - Claude 自身の解釈やコメントを付け加えない
```

```markdown:/.claude/skills/review-gemini/references/review-prompt.md
## 対象範囲

`main` から分岐した後にこのブランチで加えた変更すべてをレビューしてください。コミット済み・ステージ済み・未コミットの変更、新規追加された untracked ファイル、いずれも漏らさないこと。`main` がローカルになければ `origin/main` を使ってください。

特定のファイル・パスが対象として指示されていれば、その範囲に絞ってください。指示がなければブランチ全体が対象です。

レビュー対象が空であれば「レビュー対象なし」とだけ回答して終了してください。

## レビュー観点

次の各観点で問題がないか調査して報告してください。

<ここに自分好みの観点を書く>

## 出力フォーマット

- 指摘ごとに重大度を付ける: Critical / Major / Minor
- 具体的なファイル名・行番号を含める
- 観点ごとにセクションを分ける
- 問題がない観点は省略する
- 修正時に誤って壊さないよう、意図的に良い判断がされている箇所を簡潔に列挙する
- 褒めるためではなく、保護するために書く
```

## 初期設定

これを動かすために次の設定が必要。

### Gemini CLI インストール

公式を参照してインストール。

https://github.com/google-gemini/gemini-cli#-installation

### Google Cloud CLI インストール

Gemini CLI は Web 認証 / API token / Vertex AI 経由で使える。のだが、 Web 認証は個人 tier のみ、 API token をローカルに置くのは昨今だと自殺行為、ということで Vertex AI 経由で使う。そのために `gcloud` コマンドが必要なのでインストール。

こちらも公式を参照。

https://docs.cloud.google.com/sdk/docs/install-sdk?hl=ja

### 適当な Google Cloud プロジェクトを用意

管理しやすさのためにコードレビュー用途だとか Gemini CLI だとかがわかるようなプロジェクトを用意すると良い。

### Agent Platform API 有効化

Google Cloud プロジェクトの ID を確認し、次を開く。

https://console.cloud.google.com/apis/library/aiplatform.googleapis.com?project=<PROJECT_ID>

ここで Agent Platform API を「有効化」しておく。認証のために必要。

### Application Default Credentials 設定

次のコマンドを打つ。

```
gcloud auth application-default login
```

`Google Cloud のデータの参照、編集、設定、削除` は必須なのでこれだけ認可を通す。

### Gemini CLI 設定

ここではマシン全体で使用可能とする設定を入れる。プロジェクトに限定したい場合はプロジェクトルート直下に設定を置くと良いだろう。

```sh:~/.env
GOOGLE_CLOUD_PROJECT=<PROJECT_ID>
GOOGLE_CLOUD_LOCATION=us-central1
GOOGLE_GENAI_USE_VERTEXAI=true
```

```json:~/.gemini/settings.json
{
  "model": {
    "name": "gemini-2.5-flash"
  }
}
```

3 はまだ preview なので GA されたら切り替えたほうが良い。 2.5 はまだ sycophancy などのバイアスが多い。

### 設定できているか確認

ここまでで Gemini CLI が動くようになったか確認。適当なプロジェクトのディレクトリーに移動して次をうち、結果が返ってくれば OK 。

```sh
gemini -p "README.md の中身わかる？" --allowed-tools "run_shell_command(git)"
```

## Skill 動作確認

Claude Code 上で `/review-gemini` や `/review-gemini 未コミットのもの` などと打ってみて結果が返ってくることを確認する。

プロンプトとして「このブランチの変更をレビューして」などと投げると Claude Code 自身がレビューしてくれるので、差分を確かめてみるとよいだろう。
